# Evidence Model

## Purpose

The evidence model defines how lab activity becomes reviewable engineering evidence.

## Evidence Principles

Evidence should be:

- Generated by repeatable steps
- Stored in reviewable formats
- Tied to a specific lab or accelerator
- Clear about environment and assumptions
- Free of secrets and sensitive customer data
- Suitable for engineering and compliance review

## Evidence Types

| Evidence Type | Examples |
| --- | --- |
| Change evidence | Pull requests, commit history, review comments, issue approvals. |
| Validation evidence | Test logs, CI runs, command output, generated reports. |
| Security evidence | Scan output, permission review, threat notes, dependency reports. |
| Compliance evidence | Control mapping, approval trace, exception register, audit summary. |
| Deployment evidence | Dry-run output, environment matrix, rollback validation, artifact metadata. |
| AI workflow evidence | Prompt boundary, human review record, generated summary, reviewer disposition. |

## Required Evidence Metadata

Every evidence artifact should include:

- Artifact name
- Lab name
- Date generated
- Generator or workflow name
- Inputs used
- Validation result
- Known limitations
- Reviewer notes, if applicable

## Evidence Storage

Generated evidence should be written under:

```text
evidence-output/
```

Committed sample evidence should be stored under the relevant lab:

```text
labs/<category>/<lab-name>/evidence/
```

## Compliance Disclaimer

Evidence produced by this repository may support readiness, review, or traceability. It does not represent certification, audit completion, or production operating effectiveness unless separately validated by an authorized process.
